## 4. 功能模块详细需求

### 4.1 LLM 接入与配置

#### 4.1.1 功能描述

提供统一的 LLM 接入层，支持用户自助配置本地和云端模型提供商，所有 API Key 加密存储在本地设备。

#### 4.1.2 支持的提供商

**云端提供商**

| 提供商 | 模型示例 | 协议 |
|--------|---------|------|
| OpenAI | GPT-4o, o3 | OpenAI API |
| Anthropic | Claude 系列 | Anthropic API |
| Google | Gemini 系列 | Google AI API |
| DeepSeek | DeepSeek-V3 | OpenAI Compatible |
| 智谱 AI | GLM-4 系列 | OpenAI Compatible |
| 月之暗面 | Kimi 系列 | OpenAI Compatible |
| 百川 | Baichuan 系列 | OpenAI Compatible |
| 自定义端点 | 任意 | OpenAI Compatible |

**本地提供商**

| 提供商 | 说明 |
|--------|------|
| Ollama | 本地模型管理器，支持 Llama / Qwen / Mistral 等 |
| LM Studio | 桌面端本地推理 |
| llama.cpp | 直连推理，适合高级用户 |
| MLX | Apple Silicon 优化推理 |

#### 4.1.3 模型路由策略

系统支持可配置的智能路由规则：

- **按任务复杂度路由**：简单问答走本地小模型（< 4B），复杂推理路由到云端大模型
- **按模态路由**：图片理解请求自动路由到支持多模态的模型
- **Fallback 机制**：主模型不可用时自动切换到备用模型
- **成本控制**：可设置月度 Token 用量上限告警

#### 4.1.4 用户配置 UI 需求

- 提供商列表，支持一键添加预设提供商
- API Key 输入后立即进行可用性测试
- 显示当月 Token 用量统计（仅云端）
- 支持对每个提供商设置别名和备注
- 导入/导出配置（不含 API Key）

#### 4.1.5 安全要求

- API Key **必须** 使用 OS 原生 KeyChain（iOS）或 KeyStore（Android）存储
- 内存中的 Key 使用后立即清除
- 日志系统 **必须** 对 API Key 进行脱敏处理

### 4.2 多模态对话

#### 4.2.1 功能概述

对话支持文字、图片、文件、语音四种输入方式，以及普通语音输出和实时全双工对话两种语音模式。

#### 4.2.2 文字对话

- 支持 Markdown 渲染（代码高亮、表格、数学公式）
- 支持对话历史搜索
- 支持消息复制、分支重试（Re-generate）
- 对话上下文窗口管理：自动压缩超长上下文，保留关键摘要

#### 4.2.3 图片输入（Phase 1 实现）

**输入方式**：相机拍摄、相册选取、截图粘贴、拖拽上传（桌面端）

**处理策略**：

| 处理类型 | 执行位置 | 技术方案 | 说明 |
|----------|----------|----------|------|
| OCR 文字识别 | 本地 | Apple Vision / Google ML Kit | 无需联网，隐私安全 |
| 基础图片描述 | 本地 | LLaVA / MiniCPM-V | 质量有限但可离线 |
| 高质量多模态问答 | 云端 | GPT-4V / Gemini Vision | 需联网，走用户配置的 LLM |
| 物体 / 人脸识别 | 本地 | Core ML / ML Kit | OS 原生能力 |

**图片隐私说明**：发送图片至云端 LLM 前，系统需明确提示用户，用户可选择仅使用本地处理。

#### 4.2.4 文件输入

**支持格式**：PDF、Word（.docx）、Excel（.xlsx）、TXT、CSV、Markdown

**处理流程**：
1. 本地解析文件内容，提取纯文本
2. 对超长文档（> 50页）使用本地 RAG（向量检索增强）分块处理
3. 提取结果与用户问题一并发送给 LLM

**安全说明**：文件内容不在本地持久化存储（用户未明确要求时），处理完毕后从内存清除。

#### 4.2.5 语音输入与输出（普通模式）

**语音输入流程**：
```
用户说话 → VAD 检测停顿 → 本地 STT (Whisper.cpp) → 文字 → LLM
```

**语音输出流程**：
```
LLM 输出文字 → 本地 TTS (Kokoro / Piper) → 播放
```

**技术规格**：

| 组件 | 本地方案 | 云端备选 | 延迟目标 |
|------|----------|----------|----------|
| VAD | Silero VAD | — | < 50ms |
| STT（普通话） | Whisper.cpp (small) | Google STT / Azure | < 3s |
| STT（方言） | 不支持本地 | 火山引擎 / 阿里云 | < 2s |
| TTS | Kokoro / Piper | 阿里云 CosyVoice / ElevenLabs | < 1s |

**配置项**：
- 语音唤醒词（可选，默认关闭）
- STT 语言选择
- TTS 音色选择（本地音色库 + 云端音色）
- 语音输入自动发送 vs 手动确认

#### 4.2.6 实时全双工对话模式

> **注意**：此功能完全依赖云端 API，需用户配置对应的实时对话 Key。

**支持的后端服务**：
- OpenAI Realtime API（GPT-4o）
- Gemini Live API
- 火山引擎豆包实时语音（中文优化）

**功能要求**：
- 端到端延迟 < 500ms
- 支持打断（Barge-in）：用户说话时助理立即停止播放
- 支持情绪感知（由云端模型提供）
- 通话过程中显示实时字幕
- 通话结束后自动生成文字摘要并存储在本地

**UI 设计**：实时对话模式切换为全屏沉浸式界面，展示波形动画，清晰区分于普通对话模式。

### 4.3 生成式 UI

#### 4.3.1 功能描述

LLM 响应不局限于纯文字，可以输出结构化 Schema，客户端运行时解析后渲染为对应的原生 UI 组件。组件形式灵活（非固定模板），由 LLM 结合上下文动态决定最合适的呈现方式。

#### 4.3.2 工作原理

```
用户请求
    ↓
LLM 响应 (包含 UI Schema + 文字解释)
    ↓
UI Component Registry 匹配组件类型
    ↓
Native 渲染 → 用户可与卡片交互
    ↓
交互事件触发新的 Agent 任务
```

**UI Schema 示例（商品卡片）**：
```json
{
  "ui_type": "product_card",
  "data": {
    "name": "Sony WH-1000XM5 降噪耳机",
    "price": 2499,
    "currency": "CNY",
    "rating": 4.8,
    "image_url": "...",
    "features": ["主动降噪", "30小时续航", "多点连接"],
    "buy_url": "https://..."
  },
  "actions": [
    { "label": "查看详情", "action": "open_url" },
    { "label": "加入购物车", "action": "agent_task", "task": "add_to_cart" },
    { "label": "比较价格", "action": "agent_task", "task": "compare_price" }
  ]
}
```

#### 4.3.3 内置组件库（Phase 1）

| 组件类型 | 触发场景 | 关键字段 |
|----------|----------|----------|
| `product_card` | 商品推荐 | 名称、价格、评分、图片、操作按钮 |
| `map_preview` | 地点/路线 | 起点、终点、途经点、地图截图 |
| `contact_card` | 联系人信息 | 姓名、电话、邮件、来源 |
| `calendar_event` | 日程创建 | 标题、时间、地点、参与者 |
| `flight_card` | 航班信息 | 航班号、起降时间、座位、状态 |
| `train_card` | 高铁/火车票 | 车次、时间、座位、票价 |
| `weather_card` | 天气查询 | 城市、温度、天气状况、预报 |
| `code_block` | 代码输出 | 语言、代码、可复制/可运行 |
| `task_list` | 任务清单 | 任务项、状态、截止时间 |
| `price_chart` | 价格走势 | 时间轴、价格数据、高低点标注 |

#### 4.3.4 组件扩展机制

插件可通过注册 API 向组件库添加新的组件类型：

```typescript
interface UIComponentDefinition {
  type: string;                    // 唯一标识符
  schema: JSONSchema;              // 数据结构定义
  renderer: ComponentRenderer;     // Flutter Widget 实现
  promptHint: string;              // 提示 LLM 何时使用该组件
}
```

#### 4.3.5 响应式布局要求

- 同一 Schema 在手机/平板/桌面自适应布局
- 深色模式自动适配
- 卡片支持展开/收起详细信息

#### 4.4.1 权限分级模型

### 4.5 Agent Skill 支持与沙箱运行

#### 4.5.1 功能描述

放弃自定义插件标准，完全采用 **Anthropic Agent Skill 标准** 作为扩展能力的统一规范。每个 Skill 是一个包含 `SKILL.md` 文件的目录，`SKILL.md` 以 YAML frontmatter 描述元数据，正文提供指令，可附带脚本和资源文件。应用作为 Skill 的宿主运行环境（Host），负责加载、管理和安全执行 Skill 脚本。

> **设计原则**：我们不重新定义生态标准，而是融入 Anthropic 已有的开放标准，降低 Skill 开发者的接入成本，复用现有 Skill 生态。

#### 4.5.2 Anthropic Agent Skill 标准说明

Anthropic Agent Skill 的核心载体是 **`SKILL.md` 文件**，而非 JSON Manifest。Skill 是一个目录，结构如下：

```
flight-tracker/
├── SKILL.md          ← 必需：包含 YAML frontmatter + 指令正文
├── REFERENCE.md      ← 可选：补充参考文档
└── scripts/
    └── query.py      ← 可选：可执行脚本，由 Claude 通过 bash 调用
```

**SKILL.md 标准格式**：

```markdown
---
name: flight-tracker
description: 查询航班实时状态并推送提醒。当用户询问航班信息、起降时间、延误情况时使用。
---

# 航班追踪器

## 使用方法

调用 scripts/query.py 查询航班状态：

```bash
python scripts/query.py --flight CA1234 --date 2026-03-02
```

## 注意事项

- 需要用户提供航班号（如 CA1234、MU5735）
- 日期默认为今天，可指定 YYYY-MM-DD 格式
- 延误超过 30 分钟时主动提示用户
```

**Skill 的三层加载机制**（与 Anthropic 标准完全一致）：

| 层级 | 加载时机 | 内容 | Token 消耗 |
|------|----------|------|------------|
| Level 1：元数据 | 应用启动时始终加载 | YAML frontmatter 中的 `name` 和 `description` | ~100 tokens / Skill |
| Level 2：指令 | Skill 被触发时加载 | SKILL.md 正文（工作流、最佳实践） | < 5k tokens |
| Level 3：资源与脚本 | 按需加载 | 附属 .md 文件、可执行脚本（脚本本身不进 context，只有输出进入） | 实际无上限 |

**`name` 字段规则**（遵循 Anthropic 标准）：
- 最多 64 字符
- 只能包含小写字母、数字和连字符
- 不能包含 XML 标签
- 不能包含保留词："anthropic"、"claude"

**`description` 字段规则**：
- 不能为空，最多 1024 字符
- 应同时描述「Skill 做什么」和「何时触发」，这是 Claude 自动决策是否调用该 Skill 的依据

#### 4.5.3 沙箱运行环境

Anthropic 原生 Skill 运行在**代码执行容器（VM）**中，Claude 通过 bash 调用脚本。本应用作为 Skill 宿主，需提供等效的沙箱隔离环境，确保 Skill 脚本无法逃逸到宿主系统。

**运行时方案**：

| 脚本类型 | 沙箱技术 | 隔离方式 |
|----------|----------|----------|
| Python 脚本（`.py`） | Pyodide（WASM）| WebAssembly 沙箱，标准库受限 |
| Shell 脚本（`.sh`） | 受限 bash 环境 | 命令白名单，禁止 `rm -rf`、`curl` 等危险命令 |
| JavaScript（`.js`） | QuickJS VM | 独立 JS 引擎，无 DOM、无 Node.js 原生模块 |

**Skill 脚本执行特性**（对齐 Anthropic 标准）：
- 脚本代码本身**不进入 LLM context**，只有脚本输出进入
- Claude 读取 SKILL.md 后，通过 bash 命令调用脚本
- 脚本结果以文本返回给 Claude，Claude 再处理并回复用户

**沙箱内允许的操作**（运行时强制执行）：

```
✅ 读写沙箱内隔离的临时文件目录
✅ 执行纯计算操作
✅ 读取 Skill 目录内的资源文件
✅ 返回标准输出（stdout）给 Claude

❌ 访问宿主设备文件系统（通讯录、相册等）
❌ 发起网络请求（API 端无网络访问，与 Anthropic 标准一致）
❌ 访问其他 Skill 的数据目录
❌ 执行危险系统命令（rm、chmod、网络工具等）
❌ 动态加载外部代码
```

> **注意**：根据 Anthropic 官方文档，Claude API 端的 Skill 运行环境**无网络访问**，只能使用预装的依赖包。本应用沙箱遵循此原则，网络访问默认禁止，不设"域名白名单"机制。

**资源限制**：

| 资源 | 限制 |
|------|------|
| 单次脚本执行时间 | ≤ 30 秒 |
| 内存占用 | ≤ 128MB |
| 临时文件大小 | ≤ 100MB |
| Skill 目录总大小 | ≤ 50MB |

#### 4.5.4 Skill 市场（对接已有开放市场）

不自建市场，而是适配若干主流开放 Skill 市场，聚合展示可安装的 Skill。

**市场对接策略**：

| 市场来源 | 对接方式 | 说明 |
|----------|----------|------|
| Anthropic 官方 Skill 目录 | API 对接 | 优先展示，质量最可信 |
| 社区开放 Skill 注册表（如 GitHub 托管的 index） | 定期拉取索引 | 支持社区贡献 |
| 用户自定义来源 URL | 手动添加 | 高级用户自行添加私有市场 |

**市场 UI 功能**：
- Skill 列表浏览，支持分类筛选（效率、出行、财务、生活等）
- 关键词搜索
- Skill 详情页：描述、权限列表、版本历史、作者信息
- 安装按钮，安装前触发安全扫描流程

#### 4.5.5 安装时安全扫描

Skill 安装前必须通过本地安全扫描，对照 Anthropic 官方文档中的安全建议执行检查：

```
用户点击"安装"
        ↓
1. 下载 Skill 包（zip 解压）
        ↓
2. 结构校验
   ├── 必须包含 SKILL.md
   ├── YAML frontmatter 格式合法（name / description 字段）
   ├── name 符合命名规则（小写字母+数字+连字符，≤ 64 字符）
   └── description 非空且 ≤ 1024 字符
        ↓
3. 脚本静态扫描（针对 scripts/ 目录下所有脚本）
   ├── Python：AST 解析，检测 os.system / subprocess / socket / urllib 等危险调用
   ├── Shell：检测 curl / wget / nc / rm -rf 等危险命令
   ├── 检测外部 URL 硬编码（网络访问在沙箱内被禁止，出现则告警）
   └── 检测 eval() / exec() / __import__ 等动态代码执行
        ↓
4. SKILL.md 内容扫描
   ├── 检测是否包含 prompt injection 特征（如伪装成系统指令的内容）
   └── 检测是否引用了不存在于 Skill 包内的外部资源
        ↓
5. 扫描结果展示
   ├── ✅ 通过：展示 Skill 描述和脚本文件列表，用户确认后安装
   ├── ⚠️ 警告：发现可疑但非明确违规项，说明风险后用户可选择继续
   └── ❌ 拒绝：发现明确违规，强制拒绝安装并给出具体原因
```

> 依据 Anthropic 安全建议："像对待安装软件一样对待 Skill"——扫描覆盖 SKILL.md、脚本、资源文件等所有内容，特别警惕包含外部 URL 抓取行为的 Skill。

**扫描规则开源维护**：规则集在独立开源仓库中维护，应用定期拉取更新，社区可提交规则贡献。

#### 4.5.6 Skill 运行时管理

- Skill 列表，显示已安装 Skill 及其状态（启用/禁用）
- 点击 Skill 可查看其 SKILL.md 内容和包含的脚本文件列表
- 运行日志可查：Claude 的 bash 调用记录、脚本输出、错误日志
- 一键禁用 / 卸载，卸载时清除该 Skill 的所有临时数据
- Skill 更新时重新触发安全扫描后方可生效

### 4.6 后台定时任务

#### 4.6.1 功能描述

支持在后台静默运行定时任务，到期时触发轻量 Agent 推理，决策是否发送通知或执行自动化操作。

#### 4.6.2 任务类型

| 类型 | 示例 | 触发条件 |
|------|------|----------|
| Cron 定时 | "每天 8 点报告今日天气" | 固定时间 |
| 相对时间 | "3小时后提醒我" | 相对当前时间偏移 |
| 位置触发 | "到达公司时提醒开会" | 地理围栏进入/离开 |
| 条件触发 | "当气温低于 10 度提醒加衣" | 数据阈值 |
| 事件触发 | "邮件中有关键词'紧急'时提醒" | 外部事件 webhook |
| 周期汇总 | "每周一汇总上周完成的任务" | 周期 + 汇总逻辑 |

#### 4.6.3 平台实现方案

| 平台 | 后台任务 API | 限制说明 |
|------|-------------|----------|
| iOS | `BGTaskScheduler` + `BGProcessingTask` | 系统限制执行时间，依赖用户使用规律 |
| Android | `WorkManager` + `AlarmManager` | 受厂商电池优化影响，需引导用户白名单 |
| macOS/Windows | 系统级 Cron / Task Scheduler | 无限制 |

#### 4.6.4 轻量 Agent 推理

后台任务触发时使用本地轻量模型（< 4B 参数）进行决策推理，以降低耗电量和延迟：

```
任务触发
    ↓
本地小模型快速推理（< 1s）
    ↓
判断：是否需要通知用户？是否需要执行操作？
    ↓
发送系统通知 or 执行预设操作 or 静默跳过
```

#### 4.6.5 任务管理 UI

- 任务列表，显示下次触发时间
- 任务执行历史日志
- 支持暂停/恢复/删除任务
- 任务执行结果推送通知支持快捷操作（如"稍后提醒"、"已完成"）

### 4.7 A2A 协议与 Agent 服务发现

#### 4.7.1 功能描述

实现 Google Agent-to-Agent（A2A）协议，使助理既能作为 **Host Agent** 调用外部 Agent 服务，也能作为 **Remote Agent** 被其他系统调用。内置多种服务发现机制，降低 Agent 接入门槛。

#### 4.7.2 Agent Card 规范

每个注册的 Agent 服务通过 Agent Card 描述其能力：

```json
{
  "name": "我的个人助理",
  "description": "处理日程、邮件、购物、出行相关任务",
  "url": "https://my-agent.local:8080/a2a",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": true,
    "stateTransitionHistory": true
  },
  "skills": [
    { "id": "schedule", "name": "日程管理", "description": "创建和管理日历事件" },
    { "id": "email", "name": "邮件处理", "description": "起草和发送邮件" },
    { "id": "shopping", "name": "商品搜索", "description": "搜索比较商品价格" }
  ]
}
```

#### 4.7.3 服务发现机制

**方式一：本地局域网发现（mDNS / Bonjour）**
- 自动发现同一局域网内运行的 A2A Agent 服务
- 零配置，无需手动输入地址
- 适用场景：家庭/办公室内的多设备协作

**方式二：二维码 / URL 手动添加**
- 扫描 Agent 服务提供的二维码快速添加
- 或直接输入 Agent Card URL
- 适用场景：添加公网 Agent 服务

**方式三：接入已有公开 Agent 发现目录**
- 对接已有的 A2A Agent 公开注册目录（如社区维护的 A2A Registry），不自建
- 用户可在应用内浏览、搜索公开注册的 Agent 服务
- 用户可选择将自己的 Agent Card 发布到已有目录（仅元数据，无隐私数据）
- 对接多个目录源，用户可自行添加或移除目录源 URL

#### 4.7.4 A2A 任务执行示例

```
用户: "帮我订明天北京到上海的高铁，靠窗座位"

本地 Agent（Host）
    ├─→ A2A → 铁路信息 Agent（查询班次列表）
    │              ↓ 返回班次数据
    ├── 生成式 UI 展示班次卡片（用户选择）
    │
    ├─→ A2A → 支付 Agent（处理付款）
    │              ↓ 返回订单确认
    └─→ 本地工具 → 日历 Tool（添加出行提醒）
                   └─→ 生成式 UI → 展示订单卡片
```

#### 4.7.5 安全要求

- 所有 A2A 通信使用 HTTPS（TLS 1.3+）
- Agent 认证支持 OAuth 2.0 和 API Key
- 调用外部 Agent 前展示操作预览，用户可拒绝
- 外部 Agent 返回数据在本地沙箱中处理，不直接执行代码

### 4.8 隐私与本地安全存储

#### 4.8.1 数据分类策略

| 数据类型 | 存储位置 | 加密方式 | 是否可选同步 |
|----------|----------|----------|-------------|
| API Key | OS KeyChain / KeyStore | OS 硬件加密 | ❌ 永不同步 |
| 对话历史 | 本地 SQLite | SQLCipher (AES-256) | 待定（用户控制） |
| 联系人缓存 | 本地加密 DB | AES-256-GCM | ❌ 永不同步 |
| 位置数据 | 仅内存，不持久化 | — | ❌ 永不同步 |
| 用户偏好/画像 | 本地向量 DB | AES-256 | 待定（用户控制） |
| Skill 数据 | 隔离命名空间 | AES-256 | ❌ |
| 任务定义 | 本地 SQLite | SQLCipher | 待定（用户控制） |

#### 4.8.2 发送至 LLM 前的隐私处理

- 内置本地 **PII 检测模型**（基于轻量 NER），自动识别并标记手机号、身份证号、银行卡号等敏感信息
- 用户可启用"发送前确认"模式：每次 LLM 请求前展示完整 Prompt，用户可手动脱敏后再发送
- 对话中出现的联系人姓名，默认替换为代号（如"同事A"）再发送云端模型

#### 4.8.3 数据生命周期

- 对话历史默认保留 90 天，用户可自定义
- 一键清除所有本地数据（对话、缓存、插件数据）
- 卸载 App 时所有加密数据随 App 数据一并删除

#### 4.8.4 跨设备同步（Phase 2，基于 GitHub Gist）

采用 **GitHub Gist** 作为同步中转载体，用户使用自己的 GitHub 账号授权，数据加密后存储在私有 Gist 中，Anthropic 服务器不接触任何用户数据。

**同步方案设计**：

```
设备 A                          GitHub Gist（私有）                设备 B
  │                                     │                             │
  │  加密（用户 passphrase 派生密钥）    │                             │
  ├──────────── 推送加密数据 ──────────→ │                             │
  │                                     │ ←─── 拉取加密数据 ──────────┤
  │                                     │        解密（相同 passphrase）│
```

**同步内容**（用户可分别开关每一项）：

| 数据类型 | 默认同步 | 说明 |
|----------|----------|------|
| LLM 提供商配置（不含 API Key） | ✅ | 模型路由规则、别名等 |
| 对话历史 | ❌（用户主动开启） | 加密后同步，体积较大 |
| 定时任务定义 | ✅ | 跨设备共享提醒 |
| 用户偏好设置 | ✅ | 主题、语言、功能开关 |
| 已安装 Skill 列表 | ✅ | 不含 Skill 数据，仅 ID 列表 |
| API Key | ❌ **永不同步** | 只存在 OS KeyChain/KeyStore |

**加密方案**：
- 用户设置一个独立的同步 passphrase（与 GitHub 密码无关）
- 使用 Argon2id 从 passphrase 派生加密密钥
- 数据使用 AES-256-GCM 加密后上传，GitHub 只存密文
- passphrase 不存储在任何云端，用户须妥善保管

**GitHub 授权方式**：OAuth 2.0 授权，仅申请 `gist` 权限范围（最小权限）

**冲突处理**：Last-write-wins 策略，同步前展示差异摘要，用户可选择保留哪一侧数据
